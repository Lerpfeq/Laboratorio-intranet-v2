# LERP Intranet - Project Instructions

## Visão Geral
Sistema de intranet web para o Laboratório de Engenharia de Reações Poliméricas (LERP/FEQ), desenvolvido com Next.js 14, React 18, Prisma e PostgreSQL.

## Arquitetura Técnica

### Stack
- **Frontend**: Next.js 14 (App Router), React 18, CSS puro
- **Backend**: Next.js API Routes (serverless functions)
- **Database**: PostgreSQL (Prisma ORM)
- **Autenticação**: NextAuth.js v4 com Credentials + Google OAuth
- **Storage**: AWS S3 para uploads
- **Pacotes principais**: qrcode, xlsx, bcryptjs, framer-motion

## Sistema de Autenticação

### Login
- **Email + Senha**: Credentials Provider com bcryptjs
- **Google OAuth**: GoogleProvider (allowDangerousEmailAccountLinking: true)
- Usuários devem ser **aprovados por admin** antes de acessar a plataforma
- Status: `pending` (padrão) → `approved` (após aprovação)

### Categorias de Usuários
1. **IC (Iniciação Científica)**: Apenas leitura de estoque
2. **Pos-graduando**: Entrada/saída/consulta de reagentes, resíduos, agendamento (se autorizado), seleção de amostras, removedor de BG
3. **Admin**: Acesso total + gerenciar usuários + autorizar pós-graduandos para equipamentos

- Senha: `johndoe123`
- Categoria: Admin

## Banco de Dados

### Tabelas Principais
- **User**: Usuários com status (pending/approved) e categoria (IC/Pos-graduando/Admin)
- **Reagente**: Catálogo de reagentes (nome, marca, volume, localidade, status)
- **ReagenteEntrada**: Histórico de entradas com código interno gerado automaticamente
- **ReagenteSaida**: Histórico de saídas
- **Pedido**: Pedidos de reagentes com status (pendente/aprovado/entregue)
- **Equipamento**: Equipamentos disponíveis no laboratório
- **Agendamento**: Reservas de equipamentos
- **AutorizacaoEquipamento**: Controle de quem pode agendar cada equipamento
- **RegistroResiduo**: Registro temporário de resíduos (não armazenado)

### Índices de Performance
- User: [status, category]
- Reagente: [nome, marca, status]
- ReagenteEntrada: [reagenteId, codigoInterno, dataEntrada]
- Agendamento: [equipamentoId, dataInicio, usuarioId]

## Fluxos de Negócio

### Gestão de Reagentes
1. **Entrada**: Formulário gera código interno (LERP-{timestamp}-{random}) e cria QR code
2. **Saída**: Busca por código/nome e registra quantidade retirada
3. **Consulta**: Filtros por nome, marca, localidade, status
4. **Ordem de Pedido**: Pós-graduandos e admins criam pedidos aguardando aprovação

### Agendamento de Equipamentos
- Admin designa pós-graduandos autorizados para cada equipamento
- Calendário mostra horários ocupados/disponíveis
- Apenas pós-graduandos autorizados (ou admin) podem agendar

### Funcionalidades sem Armazenamento
- **Resíduos**: Interface para registrar tipo, peso, data. Gera documento/planilha para download. Dados são zerados após download.
- **Seleção de Amostras**: Integra algoritmo Python (fornecido). Exibe resultado, permite download. Dados não são armazenados.

## Permissões

### Nível de Middleware
- Todas as rotas (exceto /login, /signup, /api/auth/*) requerem autenticação
- Usuários não aprovados (status=pending) são redirecionados ao dashboard com aviso

### Nível de API
- Verificação de autenticação (getServerSession)
- Verificação de categoria de usuário
- IC: bloqueado em POST/PATCH (criação/edição)

### Nível de UI
- Dashboard mostra cards diferentes por categoria
- Tabs e botões desabilitados conforme permissões

### Google OAuth Setup (Necessário para Produção)
1. Criar projeto no Google Cloud Console
2. Configurar OAuth 2.0 consent screen (External, In production)
3. Criar credenciais (Web application)
4. Adicionar redirect URI: `{app_url}/api/auth/callback/google`
5. Obter Client ID e Secret
6. Substituir placeholders em .env:
   - GOOGLE_CLIENT_ID={seu_client_id}
   - GOOGLE_CLIENT_SECRET={seu_client_secret}

## Variáveis de Ambiente

### Obrigatórias
- `DATABASE_URL`: PostgreSQL connection string
- `NEXTAUTH_SECRET`: Secret para JWT (gerado automaticamente)
- `NEXTAUTH_URL`: URL da aplicação

### Opcionais (Google OAuth)
- `GOOGLE_CLIENT_ID`: Google OAuth Client ID
- `GOOGLE_CLIENT_SECRET`: Google OAuth Client Secret

### Opcionais (AWS S3)
- `AWS_BUCKET_NAME`: S3 bucket name
- `AWS_FOLDER_PREFIX`: Prefix para pastas no S3
- `AWS_REGION`: AWS region (default: us-east-1)

## Padrões de Código

### Client vs Server Components
- Páginas são **server components** por padrão
- Componentes com `useState`, `useEffect`, `useSession` têm directive `'use client'`
- Formulários e modais são client components
- API routes usam `getServerSession(authOptions)`

### Tratamento de Erros
- Try/catch em todas as funções async
- NextResponse.json para erros de API
- Mensagens de erro amigáveis no frontend
- Console.error para debug em desenvolvimento

### Crash Prevention
- Optional chaining (?.) em todos os acessos a propriedades
- Nullish coalescing (??) para valores padrão
- Validação de entrada em todas as APIs
